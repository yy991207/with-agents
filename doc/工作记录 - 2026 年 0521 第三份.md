## 六、关键架构决策回顾

按 spec §3.2 顺序:

1. **无中心调度器,任务编排器代替**:`TaskManager` 是 orchestrator 不是 dispatcher,用户做选择,不是 LLM 互相协商
2. **每轮一个独立 task,状态全在 MongoDB**:抗刷新和持久化基础
3. **SSE + HTTP/2 通信,不用 WebSocket**:think-then-choose 模式下 client→server 都是请求-响应类
4. **think 用非流式,reply 用流式**:think 50 字等齐推 UI 简单,reply 必须流式
5. **超时只在 HTTP 层**:不加业务硬超时,LLM 失败 → 卡片标失败,不阻塞其他
6. **服务端任务可恢复但不主动续推**:前端断线重连用 `GET /sse/{task_id}` + snapshot 帧
7. **agents 配置存 MongoDB 不存 yaml**(M1 决策):前端可改 + 热替换不重启 + yaml 段降为种子默认值

## 八、异常处理手法

| 场景 | 处理 |
|---|---|
| 4 个 think 全失败 | task → THINK_DONE,available_agents 为空,前端只能 regenerate / 取消 |
| think 超 50 字 | 截断到 80 字保留弹性 |
| LLM 网络超时 | httpx 层抛错被 task_manager 捕获 → think.failed 事件,不阻塞其他 agent |
| 用户 decide 时 task 不在 THINK_DONE | 路由层 KeyError → 409 Conflict |
| 双击保存配置 | TaskManager 内部 asyncio.Lock 保证只接受第一个 |
| 改 prompt 后热替换失败 | DB 已 upsert 后 reload 异常 → 再 upsert 旧值,version 连跳 +2 让客户端可观测 |
| MongoDB 短暂不可用 | 启动期连不上拒绝服务(早暴露),运行期写失败抛异常 |
| 服务进程重启时有未完成 task | 启动时未实现扫描重置,M5 待做 |
| SSE 断线 | EventSource 自动重连,但 404/410 视为致命终止 |
