# 工作记录 - 2026 年 0521 第二份

## 改动背景

本轮收到产品需求 要求 4 个 agent 的配置(model + prompt)从静态 `config.yaml` 改为可在前端动态编辑 实时生效 不重启服务。原方案中 agents 段写死在 yaml 里 前端只能展示不能改 不满足"调 prompt 不停机"的需求 因此把 agents 配置整体下沉到 MongoDB yaml 退化为种子默认值。

## 主要修改点

1. **新增 `agents` collection 持久化运行时配置**:替代 yaml 静态段 存 4 个 agent 的 model + prompt + version + updated_at name 字段唯一索引 judge"哪个 agent 当裁判"以指针形式存(可能是 agents 集合中的特殊文档 或独立 settings 集合 由实施阶段决定)
2. **yaml 的 `agents` 段和 `judge` 段降级为种子默认值**:运行时**不再读** 启动时若 DB 中 agents collection 为空才一次性 seed 注入 之后改 yaml 不生效 必须走 PUT API
3. **新增 API**:
   - `GET /api/agents` 列出所有 agent 配置(不含 key)
   - `PUT /api/agents/{name} {model?, prompt?}` 更新某个 agent 服务端原子热替换 deep_agent 实例 返回 `{ok, version}`
4. **deep_agent 实例热替换机制**:PUT 后 storage 写 DB(version + 1) 后台异步重建该 agent 的 think + reply 两个 deep_agent 实例 加锁 swap 进 `app.state.deep_agents` 字典 不重启服务 替换瞬间正在跑的 reply 用 closure 引用的旧实例不受影响
5. **前端新增 SettingsDrawer**:用于编辑 4 个 agent 的 model 和 prompt 保存后调 PUT API 收到 ok 弹 toast"已生效 当前版本 N" 失败 toast 报错不进 swap
6. **并行 team 模式拆分**:Wave 1 后端基建(M1.A)+ 文档同步(本份)Wave 2 CRUD API + 热替换(M1.C)+ 前端 SettingsDrawer(M1.D)+ 整合验证(M1 整合)

## 异常处理手法

- **DB 不可达启动拒绝服务**:沿用 §10.3 已有策略 启动时连不上 mongo 直接退出 不裸跑 早暴露问题
- **PUT 失败回滚**:storage 写失败抛异常 API 层捕获返回 5xx 前端 toast 报错 后台不进入实例 swap 阶段 内存里仍是旧实例 不会出现"DB 是新的 内存是旧的"半截状态
- **实例 swap 期间已有 reply 仍用旧实例完成**:reply 协程已 closure 引用旧 deep_agent swap 只动 dict 不打断在跑请求 不做"取消已跑 reply 让它用新版"的强一致语义(不值这个复杂度)
- **首次启动 seed 防重**:agents collection 加 `name` 唯一索引 即便并发启动两个进程也只有一个能成功 insertMany 另一个 duplicate key 跳过

## 可能影响范围

- 启动顺序变长 多了 ensure_indexes 和 seed 两步 总耗时增加但仍在秒级
- 配置真实值不再在源码里 排查问题要查 DB 而不是 grep yaml 后续运维要更新文档
- 部署时必须先把 mongo 起好 否则后端启不来(沿用 §10.3 不可达拒绝服务)
- yaml 的两个段保留但实际不读 后期可能引起新人误以为改 yaml 能生效 在 §6.3 加了显式注释提示
